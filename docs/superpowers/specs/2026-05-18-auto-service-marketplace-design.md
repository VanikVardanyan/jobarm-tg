# Дизайн: Auto Service Marketplace (Telegram-бот + Mini App)

Дата: 2026-05-18
Статус: зафиксирован, готов к написанию плана реализации
Источник требований: `SPEC_AUTO.md` (корень репо) + решения брейнсторминга ниже

## 1. Контекст и решение

Текущий репозиторий — **JobArm** (биржа услуг «клиенты ↔ мастера/отклики»). Проект
полностью разворачивается в **Auto Service Marketplace** — «клиенты ↔ автосервисы
Армении / предложения с ценой». Технологический стек SPEC_AUTO почти полностью
совпадает с текущим, поэтому **переделываем репозиторий на месте**: сохраняем
инфраструктуру и плумбинг, переписываем доменный слой, удаляем всё специфичное для
JobArm. Та же git-история, бэкап старого кода не нужен (решение пользователя).

На MVP всё бесплатно, монетизации нет. Цель — проверить, что люди этим пользуются.

## 2. Зафиксированные решения брейнсторминга

1. **Стратегия:** переделка репо на месте, инфраструктура сохраняется.
2. **Бэкап старого JobArm:** не нужен.
3. **Объём:** проектируем весь MVP сразу, реализация поэтапно (фазы 0–6).
4. **Деплой:** локально сейчас; новый бот/домен/Caddy — вне MVP-дизайна. Dev
   `docker-compose.yml` должен оставаться рабочим.
5. **Создание заявки, регистрация сервиса, отправка предложения, отзыв —
   бот-визарды в чате.** Медиа (фото/голос) обрабатываются нативно в чате, узел
   `file_id`-хендоффа из спеки устраняется. Mini App — поверхность просмотра,
   управления и выбора.
6. **Языки:** русский + армянский с самого старта (i18n с двумя локалями).
7. **Показ медиа в Mini App:** file-proxy на бэкенде (`GET /api/files/:fileId`),
   без внешнего хранилища.

## 3. Технологический стек

Без изменений относительно текущего репо:

- Backend: Node 20 + TypeScript (strict, без `any`), Fastify, grammY, PostgreSQL,
  Prisma ORM, Zod, Pino, Redis (сессии визардов + rate-limit + кэш file path).
- Frontend (Mini App): React + TypeScript + Vite, `@telegram-apps/sdk-react`,
  Tailwind CSS, React Hook Form + Zod, TanStack Query.
- Хранилище медиа: только Telegram `file_id` (без S3/диска).
- Фоновые задачи: `node-cron` в процессе сервера (одна инстанция на MVP).
- Инфраструктура: pnpm-монорепо, Docker (dev compose рабочий), деплой позже.

## 5. Модель данных (Prisma)

Берётся из SPEC_AUTO без структурных изменений. Ключевые сущности:

- `User` — и клиенты, и сервисы. `telegramId BigInt @unique`, `role CLIENT|SERVICE`,
  `language` (`ru`|`hy`), профильные поля, `isBanned Boolean @default(false)`.
- `ServiceProfile` — `userId @unique`, name, description, address, district,
  phoneNumber, lat/lng, `specializations String[]` (значения `ServiceType`),
  workingHours `Json?`, `isVerified` (модерация), `isActive`, `photos String[]`
  (Telegram file_id).
- `Car` — гараж клиента: make, model, year, bodyType?, color?, licensePlate?.
- `Request` — clientId, carId, `serviceType`, description, `voiceFileId?`,
  `photos String[]`, district, `urgency`, `isDrivable`, `status`,
  `selectedOfferId? @unique`, createdAt, `expiresAt` (createdAt + 24ч).
- `Offer` — requestId, serviceId (User), `price Int` (драмы), `priceMax Int?`,
  comment?, duration?, `examplePhotos String[]`, `@@unique([requestId, serviceId])`.
- `Review` — requestId, authorId, serviceId, `rating Int` (1–5), text?,
  `photos String[]`.

Enums: `UserRole {CLIENT, SERVICE}`, `ServiceType {BODY_PAINT, ENGINE_CHASSIS,
MAINTENANCE, TIRES, ELECTRICAL, AC, GLASS, INTERIOR, OTHER}`, `Urgency {URGENT,
THIS_WEEK, NORMAL}`, `RequestStatus {OPEN, IN_PROGRESS, COMPLETED, CANCELLED,
EXPIRED}`.

Дополнение к спеке: `User.isBanned` для админ-блокировки (спека требует
«блокировать пользователей и сервисы», но поля в схеме не было).

## 6. Медиа: Telegram file_id + file-proxy

- Все фото/голос хранятся как Telegram `file_id` (строки в `String[]`/`String?`).
- Показ в Mini App: `GET /api/files/:fileId` → бэкенд вызывает Telegram `getFile`,
  кэширует `file_path` в Redis (TTL ~50 мин, т.к. ссылка живёт ~1ч), отвечает
  `302` на актуальный CDN-URL. В Mini App — обычный `<img src="/api/files/:id">`,
  голос — `<audio>`.
- Доступ: `file_id` длинный и неугадываемый, эндпоинт публичный, но
  rate-limited. Принятый риск MVP; пост-MVP — подписанный короткоживущий токен.
- Бот при необходимости пересылает медиа другой стороне через
  `sendPhoto`/`sendVoice(file_id)` без скачивания.

## 7. Бот: меню и визарды (grammY)

Состояние визардов — сессия в Redis (grammY conversations или лёгкий FSM).

### Команды

`/start`, `/start service`, `/help`, `/cancel` (сброс визарда), `/language`.

### Онбординг

`/start` без роли → выбор: «Я ищу автосервис» (CLIENT) / «Я автосервис»
(SERVICE). Роль фиксируется на `User`.

### Меню клиента

`🚗 Создать заявку` (визард), `📋 Мои заявки` (Mini App), `🚙 Мои машины`
(Mini App, гараж), `❓ Помощь`.

### Меню сервиса

`📥 Доступные заявки` (Mini App), `📤 Мои предложения` (Mini App),
`🔧 Текущие заказы` (Mini App), `⚙️ Мой профиль` (Mini App).
До одобрения модерацией — сообщение «На модерации», меню ограничено.

### Визард: создание заявки (клиент)

1. Выбор машины: inline-кнопки из гаража; «➕ Добавить машину» → бот собирает
   make/model/year (и опц. bodyType/color/plate), создаёт `Car`.
2. Тип услуги (`ServiceType`) — inline-кнопки.
3. Описание проблемы — текст.
4. Фото — 1–5 штук (присылаются в чат, копим `file_id`); кнопка «Готово».
5. Голосовое (опционально) — voice message → `voiceFileId`.
6. Район Еревана — inline-кнопки (справочник).
7. Срочность (`Urgency`) + «машина на ходу» (`isDrivable`).
8. Подтверждение → создаётся `Request` (`OPEN`, `expiresAt = now + 24ч`).
9. Запуск матчинга (раздел 11) и сообщение «Заявка отправлена N сервисам».

### Визард: регистрация сервиса

Название → описание → адрес → район → телефон → специализации
(мультивыбор `ServiceType`) → фото мастерской (опц.) → создаётся
`ServiceProfile` с `isVerified=false`. Уведомление админам на модерацию.

### Визард: отправка предложения (сервис)

Триггер — inline-кнопка «Отправить предложение» из пуша или из Mini App
(deep link открывает чат с ботом и стартует визард по `requestId`).
Цена (или диапазон `price`/`priceMax`) → срок (`duration`) → комментарий
(опц.) → фото примеров работ (опц.) → создаётся `Offer`. Ограничение
`@@unique[requestId, serviceId]`: одно предложение от сервиса на заявку;
повторный визард **редактирует существующее** предложение (upsert), а не
создаёт второе. Редактирование запрещено, если заявка уже не `OPEN`.

### Визард: отзыв (клиент)

Через 3 дня после выбора бот шлёт запрос: inline `⭐1–5`. После оценки —
опц. «Добавить комментарий/фото». Создаётся `Review`, пересчёт рейтинга
сервиса.

### Уведомления (из бэкенда через бот)

Клиенту: новое предложение; напоминание о выборе через 6ч; запрос отзыва
через 3 дня. Сервису: новая заявка в его районе/специализации; клиент выбрал;
новый отзыв; одобрение/отклонение модерации.

## 8. API (Mini App, авторизация — Telegram initData)

Создание заявки/предложения/отзыва и регистрация сервиса — **в боте**, не в API.
Mini App API — преимущественно чтение, выбор и управление.

```
GET  /api/me                          — профиль
PUT  /api/me                          — язык, телефон

GET  /api/cars                        — гараж
POST /api/cars                        — добавить машину (через Mini App)
PUT  /api/cars/:id                    — обновить
DELETE /api/cars/:id                  — удалить

GET  /api/requests                    — мои заявки (клиент)
GET  /api/requests/:id                — детали заявки
GET  /api/requests/:id/offers         — предложения по заявке
POST /api/requests/:id/select-offer   — выбрать предложение (раскрытие контактов)
POST /api/requests/:id/cancel         — отменить

GET  /api/service/profile             — профиль сервиса
PUT  /api/service/profile             — обновить (некритичные поля)
GET  /api/service/requests            — доступные заявки (фильтр район+спец.)
GET  /api/service/requests/:id        — детали заявки
GET  /api/service/my-offers           — мои предложения + статусы
GET  /api/service/active-jobs         — где меня выбрали + контакты клиента
GET  /api/service/reviews             — отзывы обо мне

GET  /api/reviews?serviceId=          — отзывы сервиса (для карточки)

GET  /api/files/:fileId               — proxy Telegram-медиа (302 на CDN)

GET  /api/admin/services/pending      — сервисы на модерации (admin)
POST /api/admin/services/:id/verify   — одобрить/отклонить (admin)
GET  /api/admin/requests              — все заявки/предложения (admin)
POST /api/admin/users/:id/ban         — блокировка (admin)
```

Справочники (районы, типы услуг, марки/модели машин) — **статические константы
в `packages/shared`**, импортируются Mini App напрямую (workspace), отдельных
API-эндпоинтов нет.

## 9. Mini App: экраны

Mobile-first, Telegram theme colors, крупные кнопки/шрифты, минимум полей.

Клиент: **Мои заявки** (карточки со статусом) · **Детали заявки** (описание,
фото, голос, список предложений) · **Карточка предложения** (логотип/фото
сервиса, цена, срок, рейтинг, комментарий, «Выбрать») · **Гараж** (CRUD машин)
· **Профиль** (телефон, язык).

Сервис: **Доступные заявки** · **Детали заявки** (фото/описание + кнопка
«Отправить предложение» → deep link в бот) · **Мои предложения** (статусы:
ждёт/принято/отклонено) · **Текущие заказы** (контакты клиента) · **Профиль
сервиса** (редактирование) · **Отзывы**.

Админ (под `ADMIN_TELEGRAM_IDS`): **Модерация сервисов** (одобрить/отклонить),
**Заявки/предложения**, **Блокировка пользователей**.

## 10. Фоновые задачи (node-cron)

- Каждые N минут: `Request.status=OPEN && expiresAt < now` → `EXPIRED`.
- Через 6ч после первого `Offer` по заявке и при отсутствии выбора —
  напоминание клиенту (idempotent-флаг, чтобы не слать повторно).
- Через 3 дня после `select-offer` — запрос отзыва (если отзыва ещё нет).

## 11. Матчинг заявки и сервисов

При создании `Request` найти `ServiceProfile` где `isVerified && isActive &&
district == request.district && specializations ⊇ request.serviceType`.
Отправить пуш каждому: inline «Открыть в Mini App» (deep link на детали) и
«Отправить предложение» (deep link, стартует визард предложения в боте).

## 12. Локализация

- i18n с двумя локалями: `ru` (основная) и `hy` (армянский) с самого старта.
- Язык определяется из Telegram `language_code`, переключается `/language` и в
  профиле Mini App. Армянский шрифт подключается в Mini App.
- Все пользовательские тексты (бот и Mini App) — через i18n; идентификаторы и
  код — на английском.

## 13. Защита, валидация, лимиты

- Все API проверяют Telegram initData (HMAC) через существующий `plugins/auth`.
- Заблокированные пользователи (`isBanned`) отклоняются в auth/боте.
- Rate-limit (Redis-счётчики): ≤10 заявок/день/клиент, ≤50 предложений/день/сервис.
- Валидация всех входных данных через Zod (shared-схемы).
- `/api/files` — публичный, но rate-limited; доступ по неугадываемому `file_id`.

## 14. Админка

Mini App-страницы под `ADMIN_TELEGRAM_IDS` (улучшение против «basic auth /admin»
из спеки — переиспользуем существующий безопасный паттерн): модерация сервисов
(одобрить/отклонить + уведомление сервису), просмотр заявок/предложений,
блокировка пользователей/сервисов.

## 15. Поэтапная реализация

Каждая фаза — отдельный цикл план→исполнение.

| Фаза | Содержание | Спека |
|---|---|---|
| **0. Reset** | Снос домена JobArm, новая Prisma-схема + миграция, seed справочников и тестовых юзеров, shared-типы/Zod, i18n RU+HY каркас | этап 1 |
| **1. Bot core** | `/start`, выбор роли, меню клиента/сервиса, визард регистрации сервиса + статус модерации, `GET/PUT /api/me`, гейт `isBanned` | этапы 1/3 |
| **2. Клиент** | Гараж CRUD (Mini App), визард создания заявки (фото/голос), «Мои заявки» + «Детали заявки» в Mini App, rate-limit заявок | этап 2 |
| **3. Сервис** | Матчинг+уведомление сервисов, «Доступные заявки» (Mini App + deep link), визард предложения, «Мои предложения»/«Текущие заказы», rate-limit предложений | этап 3 |
| **4. Связка** | Список предложений в заявке, выбор → раскрытие контактов, уведомления обеих сторон, file-proxy медиа в Mini App | этап 4 |
| **5. Отзывы + cron** | Визард отзыва + страница «Отзывы», node-cron (expire / напоминание 6ч / запрос отзыва 3д), рейтинг сервиса | этап 4/6 |
| **6. Полировка** | Admin-модерация (Mini App), голос end-to-end, полное RU/HY покрытие, edge-cases (отмены, ошибки, истечение) | этап 5/6 |

Деплой (новый бот/домен/Caddy/prod-compose) — вне MVP-дизайна.

## 16. Критерии готовности MVP

- Клиент регистрируется и добавляет машину.
- Клиент создаёт заявку с фото и описанием через бот-визард.
- Автосервис регистрируется (с модерацией админом).
- Сервис получает уведомление о новой заявке в его районе/специализации.
- Сервис отправляет предложение через бот-визард.
- Клиент видит все предложения в Mini App и выбирает одно.
- После выбора раскрываются контакты сервиса.
- Клиент оставляет отзыв.
- Работает на русском и армянском.
- Запускается локально через dev `docker-compose`.

## 17. Что НЕ делаем на MVP

- Внутренний чат клиент↔сервис (контакты раскрываются сразу).
- Платежи, подписки, тарифы.
- AI-оценка фото.
- Эвакуатор / SOS.
- Внешнее облачное хранилище (только Telegram `file_id`).
- Сложная админка (минимум для модерации).
- Деплой на сервер (отдельно, после локальной проверки гипотезы).

## 18. Открытые риски

- `GET /api/files` без авторизации — принятый риск MVP (file_id неугадываем);
  пост-MVP — подписанные токены.
- Одна инстанция сервера для node-cron — на MVP приемлемо; при масштабировании
  переход на BullMQ/лидер-выбор.
- Истечение Telegram CDN-URL (~1ч) закрыто кэшем file_path + повторным `getFile`.
